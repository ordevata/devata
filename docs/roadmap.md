# DEVATA Roadmap — Online Booking & Communications

Дата обновления: 2025-08-27

## Обзор
Этот документ фиксирует первую фазу реализации DEVATA: запуск собственной системы онлайн-записи без сторонних сервисов, поддержку частичной предоплаты и автоматические уведомления через email, Telegram и WhatsApp (через наш n8n). План адаптирован под текущие возможности команды и инфраструктуры.

## Этап 0. Базовые принципы
- **Доменная схема:** витрина остаётся на `devata.ru`. Новые подсистемы располагаем на `app.devata.ru` (фронтенд, Vercel) и `api.devata.ru` (backend на VPS).
- **Технологический стек:** Next.js (App Router) для фронтенда, NestJS/FastAPI (можно временно Express) для API, PostgreSQL и Redis в Docker-контейнерах, оркестрация уведомлений — n8n.
- **Безопасность:** HTTPS, RBAC по ролям (partner, student, staff, branch_admin), HMAC на вебхуках, отдельные prod/stage окружения.
- **Финансовая логика:** каждую оплату делим на фонд 26% (MLM 10/3/3/1/1 + бонус 2%) и операционный фонд 74% (зарплаты, аренда, лекторы/ассистенты). Порог денежных выплат партнёрам — 50 000 ₽.

## Этап 1. Инфраструктура (неделя 1)
**Цель:** подготовить окружения для API и баз данных.
- Настроить DNS: `app.devata.ru` (CNAME → Vercel), `api.devata.ru` (A → VPS).
- Поднять на VPS (Ubuntu 22.04, 2 vCPU / 4 GB RAM) Docker + docker compose.
- Запустить базовый стек (`Caddy` reverse proxy c авто-HTTPS, `PostgreSQL`, `Redis`, демо-API) командой `docker compose up -d`.
- Настроить бэкапы БД (`pg_dump` + cron) и UFW (22/80/443).
- Проверка готовности: `https://api.devata.ru/healthz` отвечает JSON `{ "ok": true }`.

## Этап 2. Онлайн-запись без оплаты (неделя 2)
**Функциональность для клиента:** выбор центра → услуги → специалиста → свободного слота → запись.
**Функциональность для специалиста:** просмотр «Мои визиты», отметка посещения, назначение следующего визита, управление расписанием.
- **Бэкенд:** реализовать таблицы `centers`, `services`, `specialists`, `schedules`, `exceptions`, `slots`, `bookings` (идемпотентная бронь с UNIQUE по `specialist_id` + временному интервалу). Эндпоинты `/booking/slots`, `/booking` (создать/отменить/подтвердить), `/booking/follow-up`, `/specialists/{id}/schedule` и `/specialists/{id}/exceptions`.
- **Фронтенд:** страницы выбора услуги и календаря, форма данных клиента, кабинет специалиста.
- **n8n:** webhook `booking.created` → email (SMTP) + Telegram; шаблоны подтверждения, переносов, отмен.
- **DoD:** запись проходит end-to-end без оплаты; напоминания email/TG отправляются; тест гонок на один слот возвращает корректную ошибку 409 и список альтернатив.

## Этап 3. Частичная предоплата и биллинг (неделя 3)
**Цель:** добавить депозиты/остаточные платежи и отражение фонда 26%/74%.
- Политики оплаты на уровне услуги: `deposit_required`, `deposit_optional`, `full_prepaid`, `none`; параметр `deposit_percent`.
- Новые сущности: `orders`, `payments`, `funds_ledger` (фиксация 26%/74% по каждой оплате, включая депозит и остаток отдельно).
- Логика таймаутов: бронь в статусе `reserved` держит слот N минут до поступления депозита; при истечении — автоотмена и уведомление.
- Возвраты/чарджбеки: отдельные workflow в n8n, отражение отрицательных проводок.
- Фронтенд: экран оплаты депозита, статус «остаток к оплате», кнопка «Оплатить остаток».
- DoD: идемпотентные вебхуки YooKassa (`payment.succeeded`, `refund.succeeded`), корректное разделение 26%/74% в `funds_ledger`, уведомления «оплатите депозит»/«депозит получен».

## Этап 4. Каналы уведомлений (неделя 4)
**Цель:** расширить коммуникации до триады email + Telegram + WhatsApp.
- **Telegram:** бот, хранение `chat_id`, рассылка через n8n Telegram node.
- **Email:** SMTP, шаблоны в HTML, DKIM/SPF в DNS.
- **WhatsApp:** запуск собственного шлюза (Node.js + Baileys) на `wa.devata.ru`, авторизация по QR, REST API `/send` и `/wa/inbound` с HMAC. Троттлинг и «тихие часы».
- **n8n:** fallback на email/TG при недоступности WhatsApp, логирование в Google Sheets.
- DoD: уведомления по всем трём каналам при записи, переносе, отмене и напоминаниях (T-24/T-2/T-15). Автоматическое переключение при сбое одного из каналов.

## Этап 5. Админ-панель (неделя 5)
- Интерфейсы управления услугами, ценами, длительностью, депозитами.
- Настройка расписаний и исключений специалистов, лимитов бронирования (например, ≤30 дней вперёд).
- Управление политиками отмен и SLA напоминаний.
- Конфигурирование каналов уведомлений (вкл/выкл, приоритет).
- Audit-log административных действий.
- DoD: все ключевые настройки меняются через UI без правки кода; журнал аудита хранится минимум 180 дней.

## Этап 6. Финансовая отчётность (неделя 6)
- Отчёты по броням/оплатам, сводка 26%/74% за период, остатки фонда 26% (выплачено/резерв).
- Ежедневные выгрузки в Google Drive (шаблоны с Google Sheets) через n8n.
- Контрольные сверки с YooKassa (идемпотентные ключи операций).
- DoD: два отчёта в админке (день/месяц), автоматическая выгрузка в Drive, сверка оборотов с провайдером на тестовых данных.

## Этап 7. Партнёрка и обучение (неделя 7–8)
- Реализация MLM 5 уровней (10/3/3/1/1) + бонус 2% обученным специалистам, учёт порога 50 000 ₽ и возможности оплаты услуг вместо денежной выплаты.
- Кабинеты ролей (partner/student/staff/branch_admin) на базе кастомного му-плагина `devata-cabinets`.
- Учёт лекторских/ассистентских бюджетов (25%/10–15%) из фонда 74%, контроль состава групп и самоназначения.
- DoD: начисления совпадают с регламентом, партнёры видят структуру и реестры выплат, студенты — курсы и прогресс, сотрудники — расписание и выплаты.

## Сквозные процессы n8n
1. **Leads Hub:** Telegram, WhatsApp, сайт, соцсети → нормализованные лиды → CRM.
2. **Booking Intake:** Webhook API → сохранение в БД → уведомления → напоминания.
3. **Payments:** Webhooks YooKassa → расчёт фонда 26/74 → обновление `funds_ledger` → кабинеты.
4. **Notifications:** расписание напоминаний, шаблоны, fallback-логика.
5. **Backups:** выгрузка воркфлоу в GitHub и данных в Google Drive.
